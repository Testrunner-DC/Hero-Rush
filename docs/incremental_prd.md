## 5. 各模块详细需求描述

### 模块A：补全游戏引擎流程

#### A-1. 开局调度完整实现（规则书 303.1 节）

**现状分析：**

当前 `GameSetup.tsx` 已实现4步流程（洗牌→先后手→起手→调整），但存在以下缺陷：

1. **手牌调整（Mulligan）不完整**：Step 3 仅提供 +/- 按钮调整手牌数量，并非实际选择哪些牌放回卡组底。规则书 303.1.e 要求："调整的牌背面朝上放回卡组底，然后从卡组顶抽取等于调整手牌数的卡，再洗混卡组"
2. **先攻玩家先行决定**：当前 UI 未体现先攻玩家先行调整的顺序
3. **卡组洗混**：调整后需要重新洗混卡组，当前未实现
4. **快速开始模式**：`handleQuickStart` 直接跳过调整阶段，不符合规则

**需求规格：**

```
规则 303.1.e — 调整起始手牌流程：
  1. 先攻玩家先行决定是否调整
  2. 调整步骤：
     a. 玩家选择要调整的手牌（1~6张）
     b. 选中的牌背面朝上放回卡组底
     c. 从卡组顶抽取等量牌
     d. 洗混卡组
  3. 后攻玩家再行决定是否调整
  4. 双方均完成后进入游戏
```

**涉及文件：**
- `src/components/GameSetup.tsx` — 重写 Step 3 调整手牌逻辑
- `src/types/game.ts` — 可选新增 `SetupPhase` 类型扩展（如 `"MULLIGAN_P1"` / `"MULLIGAN_P2"`）

**验收标准：**
- [ ] 先攻玩家先行调整，后攻玩家再调整
- [ ] 玩家可选择具体手牌放回卡组底（非仅调整数量）
- [ ] 放回的牌进入卡组底，从卡组顶抽等量牌
- [ ] 调整后洗混卡组
- [ ] 玩家可选择不调整（直接跳过）
- [ ] 快速开始模式仍可用（但标注为简化模式）

#### A-2. 应对阶段接入引擎（规则书 303.2.a.4.3.2 + 305.1 节）

**现状分析：**

1. `PendingCounter` 类型已在 `src/types/game.ts` 中定义（`summoningPlayerIdx` / `summoningCardId` / `summoningZone`），`BattleState.pendingCounter` 字段已存在
2. SD01-011 和 SD02-004 已注册 `category: "counter"` 的效果定义，但 `condition` 检查 `state.pendingCounter != null`
3. `GameAction` 联合类型中**缺少** `TRIGGER_COUNTER` / `RESOLVE_COUNTER` / `PASS_COUNTER` 三个 action
4. 引擎 `gameReducer` 中**没有**应对阶段的处理逻辑
5. 号召流程 (`handleSummonToField`) 中**未设置** `pendingCounter`

**需求规格：**

```
应对规则（303.2.a.4.3.2 + 305.1）：
  - 触发时机：对方玩家进行号召时
  - 应对方式：玩家进行1张拥有【应对】能力的角色的号召
  - 限制：每回合双方各自最多1次
  - 结束条件：双方连续选择"不行动"则应对步骤结束
  - 应对·起动：玩家可起动其卡牌的"应对·起动"效果（如 SD01-002/016 的应对·起动效果）
```

**新增 GameAction 定义：**

```typescript
// src/game/types.ts 新增
| { type: "TRIGGER_COUNTER"; playerIdx: number; cardId: string; handIndex: number }
  // 玩家触发应对：使用手牌中拥有【应对】的角色进行号召
  
| { type: "RESOLVE_COUNTER"; playerIdx: number; effectCardId: string }
  // 玩家使用应对·起动效果（如拦截）
  
| { type: "PASS_COUNTER"; playerIdx: number }
  // 玩家选择不行动
```

**引擎处理逻辑：**

1. **号召时检查应对窗口**：在 `handleSummonToField` / `handleSelectRetreat` 完成号召后，设置 `pendingCounter`：
   ```
   pendingCounter = {
     summoningPlayerIdx: 发起号召的玩家,
     summoningCardId: 被号召的卡牌ID,
     summoningZone: 号召目标区域
   }
   ```
2. **应对状态追踪**：新增 `counterUsedThisTurn: [boolean, boolean]` 到 `BattleState`，追踪本回合双方是否已使用应对
3. **应对执行**：`TRIGGER_COUNTER` 时，检查对方手牌中是否有【应对】角色，执行号召（作为应对方式的号召）
4. **应对·起动**：`RESOLVE_COUNTER` 时，查找场上/手牌中的应对·起动效果并执行
5. **不行动检查**：`PASS_COUNTER` 后，若双方连续选择不行动，清除 `pendingCounter`，继续原流程
6. **回合结束重置**：`handleEndTurn` 中重置 `counterUsedThisTurn`

**涉及文件：**
- `src/game/types.ts` — 新增3个 GameAction
- `src/types/game.ts` — 新增 `counterUsedThisTurn` 字段到 `BattleState`
- `src/game/engine.ts` — 新增 `handleTriggerCounter` / `handleResolveCounter` / `handlePassCounter`；修改 `handleSummonToField` 和 `handleSelectRetreat` 设置 `pendingCounter`
- `src/pages/BattlePage.tsx` — UI 层显示应对窗口

**验收标准：**
- [ ] 对方号召时，若我方手牌有【应对】角色，显示应对提示
- [ ] 玩家可选择触发应对（号召【应对】角色）或不行动
- [ ] 玩家可选择使用应对·起动效果（如有）
- [ ] 每回合双方各自最多1次应对
- [ ] 双方连续不行动后应对步骤结束，继续原流程
- [ ] 回合结束时重置应对计数

4. 当前引擎 `handleAdvancePhase` 中 ACTION 阶段没有"起动效果"的入口
5. 起动效果的来源区域 (`activeSource: "hand" | "base" | "field"`) 已定义但未在引擎中使用

**需求规格：**

```
规则 303.2.a.3.1.5 + 304.2：
  - 行动阶段可起动其卡牌的起动效果和应对·起动效果
  - 起动型效果一旦起动，立刻结算
  - 若效果文本存在"可以结算的内容"，玩家才能起动
  - 若含有"如此做后"，需在"如此做后"之前存在可结算情况

新增 GameAction：
  | { type: "ACTIVATE_EFFECT"; playerIdx: number; cardId: string; effectId?: string }
  
引擎处理：
  1. 验证：活跃玩家 + ACTION 阶段
  2. 查找卡牌的 active 类型效果（通过 card_no 查注册表）
  3. 检查 activeSource（手牌/基地/场上）与卡牌当前区域匹配
  4. 检查 condition 和 cost
  5. 执行 execute
  6. 若 faceDownAfterActive=true，将卡牌盖伏（如 SD02-005/008/009）
  7. 若效果有 once 标记，标记已使用
```

**涉及文件：**
- `src/game/types.ts` — 新增 `ACTIVATE_EFFECT` action
- `src/game/engine.ts` — 新增 `handleActivateEffect` 函数
- `src/types/game.ts` — 新增 `activatedEffectsThisTurn: string[]` 追踪本回合已起动的效果
- `src/pages/BattlePage.tsx` — UI 层显示可起动的效果按钮

**验收标准：**
- [ ] 行动阶段可起动手牌/基地/场上的起动效果
- [ ] 起动效果检查 condition 和 cost
- [ ] 执行后若有 `faceDownAfterActive` 则盖伏卡牌
- [ ] `once` 标记的效果本回合不可重复起动
- [ ] UI 层高亮显示可起动的卡牌和效果

## 7. 技术约束与注意事项

1. **纯函数 reducer 模式**：所有引擎修改必须保持 `gameReducer` 为纯函数，不可有副作用。卡牌效果通过 `EffectContext` 传递状态，返回新 `BattleState`。

2. **不可变更新**：所有状态更新使用展开运算符（`{...state}`），不直接修改原对象。

3. **效果注册幂等性**：`registerAllEffects` 已有 `effectsInitialized` 标志防止重复注册。新增效果需在对应 sd01.ts/sd02.ts 中追加到 `SD01_EFFECTS` / `SD02_EFFECTS` 数组。

4. **TypeScript 类型安全**：新增 GameAction 需更新 `src/game/types.ts` 中的联合类型；新增 BattleState 字段需更新 `src/types/game.ts`。

5. **构建验证**：修改后需执行 `npm run build` 确认 TypeScript 0 错误、Vite 构建成功。

6. **EDITOR_GUIDE.md 的受众**：文档面向完全不懂代码的项目经理，所有技术术语需附带通俗解释，操作步骤需精确到"打开哪个文件、找到哪一行、改成什么"。
