## 一、产品定位

### 1.1 核心价值

为**漫威超英击战（Marvel TCG）**玩家提供：
1. **自动化对战**：完整复刻综合规则书1.00的全部规则（9区域、6阶段回合、Lv号召系统、R值攻击距离、冲刺卡时间线胜利机制、6大能力关键词）
2. **卡查数据库**：可搜索的角色卡+冲刺卡数据库，支持多维度筛选
3. **卡组构筑**：50张角色卡（最多2色，同名≤3张）+ 9张冲刺卡的自组与合法性检查
4. **赛事系统**：瑞士轮/单淘汰赛事全生命周期管理

### 1.2 竞品参考

| 平台 | 对战 | 卡查 | 赛事 | 卡组 | 特点 |
|------|------|------|------|------|------|
| mobilesuitarena.com | ✅ | ✅ | ✅ | ✅ | 高达卡牌，参考蓝本 |
| Untap.in | ✅ | ✅ | ❌ | ✅ | 通用TCG，规则简单 |
| Cockatrice | ✅ | ✅ | ❌ | ✅ | 开源，手动结算 |
| **本项目目标** | **✅ 自动结算** | **✅** | **✅** | **✅** | **超英击战专属** |

### 1.3 游戏规则核心要素（综合规则书1.00）

| 规则要素 | 说明 | 引擎实现难度 |
|---------|------|-------------|
| 卡组构筑 | 50张角色卡（≤2色，同名≤3）+ 9张冲刺卡 | ⭐⭐ |
| 9个游戏区域 | 先锋(1)/侧翼(4)/后卫(1)/基地区(6)/撤退/虚空/时间线/卡组/冲刺卡组 | ⭐⭐⭐ |
| 3种胜利方式 | 时间线满9张冲刺卡 / 对方卡组耗尽 / 卡牌效果宣布胜利 | ⭐⭐ |
| 6阶段回合 | 开始→抽2→行动(基地部署/号召≤3/战基移动/起动)→战斗→应对→结束 | ⭐⭐⭐⭐ |
| Lv号召系统 | Lv3及以下直接上场；Lv4+需撤退场上总Lv≥角色Lv的卡（盖卡算Lv1） | ⭐⭐⭐⭐ |
| R值攻击距离 | R-1相邻/R-2隔1格/R-3远程，决定可攻击区域 | ⭐⭐⭐ |
| 战斗三步骤 | 目标步骤→应对步骤→判定步骤（角色vs角色/角色vs破绽） | ⭐⭐⭐⭐⭐ |
| 6大能力关键词 | 应对（手牌）、拦截（应对·起动）、连击/强袭/空袭（战区）、唯一（场上） | ⭐⭐⭐⭐ |
| 效果系统 | 触发型（条件→效果）、起动型（主动激活） | ⭐⭐⭐⭐ |
| 优先级规则 | 【不能】>【能】，卡牌拥有者/控制者区分 | ⭐⭐⭐ |
| 特殊机制 | 盖卡/翻开、结附卡、替换卡牌、战区/战基移动、舍弃/撤退/裁剪 | ⭐⭐⭐⭐ |

**可延后（P1）**：
6. 匹配系统（天梯/排位）
7. 赛事系统（瑞士轮）
8. 对战回放

**后续迭代（P2）**：

12. 赛事直播/观战

### 5.2 推荐启动路径

```
Step 1: 卡查网站先行（1-2个月）
  ↓  可独立上线，验证用户需求
Step 2: 卡组构筑器（0.5个月）
  ↓  吸引玩家使用，积累用户
Step 3: 对战引擎核心（2-3个月）
  ↓  最难的部分，建议用AI辅助加速
Step 4: 对战界面+实时对战（1-2个月）
  ↓  MVP完成，开放Beta
Step 5: 赛事系统（1个月）
  ↓  正式运营
```

**理由**：卡查+卡组构筑器可以快速上线验证需求，同时为对战引擎积累结构化卡牌数据。对战引擎是最耗时的部分（规则复杂度高），应集中精力攻坚。

## 六、风险评估

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|

| 用户量增长导致性能瓶颈 | 低 | 高 | 引擎无状态设计，可水平扩展 |
| 综合规则书更新导致规则变更 | 中 | 中 | Lua脚本化规则（专业版）/ JSON规则定义（精简版），便于热更新 |

## 七、总结与建议

### 7.1 推荐方案

**推荐方案A（精简版）**，理由：
1. 用户是独立开发者，具备Vue 3 + Python + 数据分析能力
2. 超英击战作为新游戏（规则书1.00刚发布），玩家基数尚在增长，过早投入大规模开发风险高
3. 卡查+卡组构筑器可快速上线验证需求，失败成本低
4. Token成本可控（开发期¥7,000，运营期¥480/月）

### 7.2 关键成功因素

1. **规则准确性**：对战引擎必须100%正确实现综合规则书1.00，建议建立规则测试用例库
2. **卡牌数据质量**：282张卡的结构化数据是基础，OCR+人工校对不可省略
3. **社区冷启动**：与B站@笛喜_DC等超英击战内容创作者合作推广
4. **迭代速度**：先上MVP收集反馈，再逐步完善
