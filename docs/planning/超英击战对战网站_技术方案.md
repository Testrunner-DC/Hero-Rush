## 2. 对战引擎设计

### 2.1 核心挑战分析

漫威超英击战（Marvel TCG）对战引擎是本项目的最大技术挑战，主要难点：

1. **规则复杂性**：游戏包含角色卡（50张入卡组）+ 冲刺卡（9张自组）， rule复杂度高
2. **Lv召唤系统**：Lv4+角色需要撤退场上总Lv≥该角色Lv的卡作代价（盖卡算Lv1）
3. **胜利机制**：时间线集满9张冲刺卡获胜，攻击破绽→放冲刺卡到时间线
4. **战斗系统**：R值攻击距离（R-1相邻/R-2隔1格/R-3远程）+ 破绽机制 + 关键词
5. **9区域管理**：先锋区(1)+侧翼区(4)+后卫区(1)+基地区(6)+撤退区+虚空区+时间线+卡组+冲刺卡组
6. **6阶段回合**：开始→抽2张→行动（基地部署/号召/战基移动/起动）→战斗（调位→按序攻击→目标/应对/判定）→应对→结束
7. **实时同步**：多玩家状态必须严格一致
8. **可扩展性**：新卡牌不断推出，引擎需支持快速扩展
9. **回放支持**：需记录完整操作序列用于回放

### 2.2 技术方案选型

#### 方案对比

| 方案 | 优点 | 缺点 | 适用性 |
|------|------|------|--------|
| **自研引擎** | 完全可控，针对性优化 | 开发周期长，bug风险高 | ✅ 推荐 |
| **改造Cockatrice** | 开源TCG引擎，有基础 | 代码老旧，架构耦合 | ⚠️ 可考虑 |
| **改造Forge(MTG)** | MTG开源引擎，功能强 | 规则差异大，改造成本高 | ❌ 不推荐 |
| **使用Lua脚本** | 规则可配置，热更新 | 性能需优化 | ✅ 推荐（自研+Lua） |

#### 2.3.4 Lv召唤系统

**Lv4+召唤规则实现**：

```go
// 验证号召（召唤）的合法性
func (a *CallAction) Validate(state *GameState) error {
    player := state.GetActivePlayer()
    card := state.GetCard(a.CardInstanceID)
    
    // 检查Lv等级
    if card.Level >= 4 {
        // 计算撤退卡的Total Lv
        totalLv := 0
        for _, retreatCardID := range a.RetreatCards {
            retreatCard := state.GetCard(retreatCardID)
            if retreatCard.IsFaceDown {
                totalLv += 1 // 盖卡算Lv1
            } else {
                totalLv += retreatCard.Level
            }
        }
        
        // 检查Total Lv是否≥角色Lv
        if totalLv < card.Level {
            return fmt.Errorf("insufficient Lv: need %d, have %d", card.Level, totalLv)
        }
    }
    
    return nil
}
```

#### 2.3.5 战斗系统（R值攻击距离 + 破绽机制）

**R值攻击距离实现**：

```go
// 检查攻击是否合法（基于R值）
func (a *AttackAction) Validate(state *GameState) error {
    attacker := state.GetCard(a.AttackerID)
    target := state.GetCard(a.TargetID)
    
    // 计算攻击距离（基于R值）
    distance := state.CalculateDistance(attacker, target)
    
    switch attacker.RValue {
    case 1: // R-1: 相邻
        if distance != 1 {
            return fmt.Errorf("R-1 attack requires adjacent target")
        }
    case 2: // R-2: 隔1格
        if distance != 2 {
            return fmt.Errorf("R-2 attack requires target 2 spaces away")
        }
    case 3: // R-3: 远程
        if distance < 3 {
            return fmt.Errorf("R-3 attack requires ranged target")
        }
    }
    
    return nil
}

// 判定阶段（破绽机制）
func (state *GameState) JudgeAttack(attacker *CardInstance, target *CardInstance) (*JudgmentResult, error) {
    // 掷骰或判定破绽
    // 攻击成功 → 放冲刺卡到时间线
    // 攻击失败 → 攻击者受伤害
    
    result := &JudgmentResult{}
    
    // 简化的判定逻辑（实际更复杂）
    if attacker.Power > target.Power {
        result.Success = true
        result.SprintCard = target.Owner.GetSprintCard() // 从冲刺卡组拿一张
        state.AddToTimeline(target.Owner, result.SprintCard)
        
        // 检查胜利条件
        if state.GetTimelineCount(target.Owner) >= 9 {
            result.GameEnd = true
            result.Winner = attacker.Owner
        }
    } else {
        result.Success = false
        attacker.Counters["damage"] += target.Power
    }
    
    return result, nil
}
```

#### 2.3.7 事件系统 (Event System)

```go
type GameEvent struct {
    ID        string      // 事件ID
    Type      EventType   // 事件类型
    Timestamp time.Time   // 时间戳
    PlayerID  string      // 触发玩家
    Data      interface{} // 事件数据
}

type EventType string

const (
    EventGameStart     EventType = "GAME_START"
    EventTurnStart     EventType = "TURN_START"
    EventDrawCards    EventType = "DRAW_CARDS"
    EventCallCharacter EventType = "CALL_CHARACTER"
    EventAttack        EventType = "ATTACK"
    EventJudgment     EventType = "JUDGMENT"
    EventAddSprint    EventType = "ADD_SPRINT"    // 放冲刺卡到时间线
    EventGameEnd      EventType = "GAME_END"
)

// 事件总线
type EventBus struct {
    subscribers map[EventType][]EventHandler
    mu          sync.RWMutex
}

func (eb *EventBus) Subscribe(eventType EventType, handler EventHandler) {
    eb.mu.Lock()
    defer eb.mu.Unlock()
    eb.subscribers[eventType] = append(eb.subscribers[eventType], handler)
}

func (eb *EventBus) Publish(event GameEvent) {
    eb.mu.RLock()
    handlers := eb.subscribers[event.Type]
    eb.mu.RUnlock()
    
    for _, handler := range handlers {
        go handler.Handle(event)  // 异步处理
    }
}
```

## 5. 实时通信方案

### 5.1 方案选型

| 方案 | 优点 | 缺点 | 适用性 |
|------|------|------|--------|
| **WebSocket** | 全双工，低延迟 | 需自行处理重连 | ✅ 推荐 |
| **Socket.io** | 自动重连，房间管理 | 额外依赖 | ✅ 推荐（开发效率） |
| **SSE** | 简单，HTTP兼容 | 单向，不支持客户端推送 | ⚠️ 仅用于观战 |
| **gRPC-Web** | 高性能，类型安全 | 复杂度高 | ❌ 不推荐 |

**推荐方案**：Socket.io（对战） + SSE（观战直播）

### 5.4 心跳与重连机制

```javascript
// 客户端心跳
const heartbeatInterval = setInterval(() => {
  socket.emit('heartbeat', { timestamp: Date.now() });
}, 30000);

// 服务器心跳检测
socket.on('heartbeat', (data) => {
  socket.data.lastHeartbeat = data.timestamp;
});

// 超时检测（60秒无心跳则断开）
setInterval(() => {
  const now = Date.now();
  for (const [socketId, socket] of io.sockets.sockets) {
    if (now - socket.data.lastHeartbeat > 60000) {
      socket.disconnect(true);
    }
  }
}, 60000);

// 客户端重连
const socket = io('/ws/battle', {
  reconnection: true,
  reconnectionAttempts: 5,
  reconnectionDelay: 1000,
  reconnectionDelayMax: 5000,
});
```

## 7. 技术风险与应对

### 7.1 高风险点识别

| 风险ID | 风险描述 | 影响等级 | 概率 | 风险值 |
|--------|---------|---------|------|--------|
| R-001 | 漫威超英击战规则复杂度超预期，引擎开发周期延长 | 高 | 高 | 9 |
| R-002 | 对战引擎性能不足，无法支持高并发 | 高 | 中 | 6 |
| R-003 | 卡牌数据获取困难，缺乏官方授权 | 中 | 高 | 6 |
| R-004 | 实时同步出现状态不一致（网络延迟/重连） | 高 | 中 | 6 |
| R-005 | 第三方登录集成复杂（微信/QQ OAuth） | 中 | 中 | 4 |
| R-006 | 赛事系统配对算法错误，影响公平性 | 高 | 低 | 3 |
| R-007 | 对战回放数据过大，存储成本高 | 中 | 中 | 4 |
| R-008 | DDoS攻击导致服务不可用 | 高 | 低 | 3 |

### 7.2 风险应对策略

#### R-001：规则复杂度超预期

**缓解方案**：

1. **分阶段开发**：先支持核心规则（80%常用卡牌），再逐步扩展
2. **规则脚本化**：使用Lua脚本，新卡牌只需添加脚本，无需改引擎代码
3. **社区协作**：开源规则脚本，让社区贡献卡牌定义
4. **参考竞品**：研究类似TCG游戏的规则实现方式

**Plan B**：如果自研引擎进度严重滞后，考虑与有TCG引擎经验的团队合作，或购买商业引擎授权。

#### R-002：对战引擎性能不足

**缓解方案**：

1. **性能测试先行**：开发同时编写性能测试，确保单引擎pod支持100+并发对战
2. **水平扩展**：引擎设计为无状态，可通过增加pod数量扩展
3. **优化热点**：识别性能瓶颈（Lua执行、状态序列化），针对性优化
4. **混合架构**：计算密集型操作（AI）使用Go实现，规则脚本用Lua

#### R-003：卡牌数据获取困难

**缓解方案**：

1. **爬虫方案**：编写爬虫从官方或社区网站获取卡牌数据
2. **OCR识别**：开发卡牌截图OCR识别功能，用户上传截图自动录入
3. **社区贡献**：建立卡牌数据库编辑系统，让玩家贡献数据
4. **法务评估**：咨询律师，确认数据使用是否构成侵权

**Plan B**：如果无法获得数据授权，考虑先做其他TCG（如万智牌）的线上平台，积累技术后再做漫威超英击战。

#### R-004：实时同步状态不一致

**缓解方案**：

1. **状态版本号**：每次状态更新携带版本号，客户端检测版本跳跃则请求完整状态
2. **服务端权威**：所有动作由服务端验证和执行，客户端只做预测显示
3. **重连恢复**：断线重连用battle_id重新加入，服务端推送当前完整状态
4. **事务日志**：记录所有动作序列，可用于状态重建

#### R-005：第三方登录集成复杂

**缓解方案**：

1. **使用成熟库**：Node.js使用Passport.js，已支持微信/QQ OAuth
2. **沙箱测试**：提前申请测试账号，在开发环境充分测试
3. **降级方案**：如果第三方登录遇到问题，优先保证邮箱注册可用

#### R-006：赛事配对算法错误

**缓解方案**：

1. **使用成熟算法**：瑞士轮使用《Swiss Pairing Algorithm》（借鉴chess.com实现）
2. **单元测试**：编写大量单元测试，覆盖各种边界情况
3. **手动调整**：提供组织者手动调整配对功能，作为自动配对的备份

#### R-007：回放数据存储成本高

**缓解方案**：

1. **增量存储**：只存储动作序列，不存储完整状态快照（除非关键节点）
2. **压缩存储**：使用gzip压缩回放JSON，减少70%存储空间
3. **分级存储**：热数据存Redis，温数据存PostgreSQL，冷数据归档到S3
4. **定期清理**：超过1年的回放数据自动归档或删除

#### R-008：DDoS攻击

**缓解方案**：

1. **CDN防护**：使用Cloudflare或阿里云CDN，过滤恶意流量
2. **速率限制**：API网关层实施速率限制（如每IP 100 req/min）
3. **WAF**：部署Web应用防火墙，拦截SQL注入、XSS等攻击
4. **弹性扩容**：Kubernetes HPA自动扩容，吸收流量冲击

## 9. 附录

### 9.1 技术决策记录（ADR）

#### ADR-001：选择Go+Lua作为对战引擎技术栈

**状态**：已接受

**背景**：
对战引擎需要高性能和规则可扩展性。纯Go性能好但规则修改需要重新编译；纯Lua易扩展但性能较差。

**决策**：
采用Go（核心引擎）+ Lua（规则脚本）混合方案。

**后果**：
- ✅ 规则可热更新，新卡牌无需重新部署
- ✅ Go保证核心性能，Lua处理规则逻辑
- ❌ 增加系统复杂度，需要维护两种语言环境

#### ADR-002：选择Socket.io而非原生WebSocket

**状态**：已接受

**背景**：
需要实时双向通信。原生WebSocket需要自行处理重连、房间管理。

**决策**：
使用Socket.io库。

**后果**：
- ✅ 开发效率高，自动重连、房间管理开箱即用
- ✅ 社区活跃，文档完善
- ❌ 增加客户端bundle体积(~30KB gzipped)
