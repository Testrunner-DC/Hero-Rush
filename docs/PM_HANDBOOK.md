## 1. 产品总览与愿景

### 1.1 产品定位

**斗界竞技场（Hero Rush）** 是超英击战（Marvel TCG）的非官方卡牌查询、组卡与对战平台。

**一句话定位**：为超英击战玩家提供从查卡、组卡到对战的完整数字体验。

### 1.2 目标用户画像

| 画像 | 描述 | 核心需求 |
|------|------|---------|
| **新手上手型** | 刚接触 TCG，需要学习规则 | 浏览卡牌、读懂效果文本、用预组体验对战 |
| **构筑研究型** | 有 TCG 经验，研究卡组构成 | 组卡器效率、多维度筛选、起手模拟、卡组导出 |
| **竞技对战型** | 深度玩家，需要实战练习 | 完整对战引擎、卡效正确性、对战速度 |
| **社交分享型** | 喜欢分享和发现新卡组 | 卡组广场、发布功能、收藏和评论 |

### 1.3 核心价值主张

1. **零门槛上手** — 纯浏览器端运行，无需安装
2. **规则实现完整** — 对战引擎实现完整规则，非"骰子模拟器"
3. **对标行业最佳** — UI/UX 对标 Piltover Archive，卡牌社区体验一致性
4. **开放可维护** — 数据驱动架构，新增卡包只需更新 JSON

### 1.4 关键指标（North Star Metrics）

| 指标 | 当前 | 目标 | 衡量方式 |
|------|------|------|---------|
| 卡组创建数 | — | 100+ | localStorage / Supabase |
| 对战总局数 | — | 500+ | 本地统计 |
| 卡效正确率 | 100% (31/31 测试) | 持续保持 | Vitest 自动化 |
| 页面加载速度 | < 2s (首屏) | < 1.5s | Lighthouse |
| 构建零错误 | ✅ | 持续保持 | CI/CD |

## 2. 团队角色与职责边界

### 2.1 角色矩阵

| 角色 | 负责人 | 核心职责 | 产出物 |
|------|--------|---------|--------|
| **产品经理（PM）** | 许清楚 | 需求分析、PRD 撰写、竞品调研 | PRD / 用户故事 |
| **架构师** | 高见远 | 系统设计、技术选型、任务分解 | 架构设计 / 任务列表 |
| **工程师** | 寇豆码 | 代码实现、全局一致性审查 | 源代码 / Code Review |
| **QA 工程师** | 严过关 | 测试编写、质量验证、Bug 追踪 | 测试报告 / Bug 清单 |
| **交付总监** | 齐活林 | 流程编排、消息中转、交付管理 | 交付总结 |

### 2.2 协作流程（SOP）

所有需求走正式团队协作流程，严禁跳过：

```
用户需求
  │
  ├── 小型需求（≤10 源文件）→ ⚡ 快速模式
  │     TeamCreate → 工程师(一次性实现) → QA(验证) → 交付
  │
  ├── Bug 修复 → 🔧 BugFix 快捷路径
  │     TeamCreate → 工程师(定位+修复) → QA(回归测试) → 交付
  │
  ├── 中大型需求（>10 源文件）→ 🏗️ 标准 SOP
  │     PM(PRD) → 架构师(设计+任务) → 工程师(实现) → QA(测试) → 交付
  │
  └── 仅需分析 → 📋 部分工作流
        PM(PRD) / 架构师(评审) / 市场调研
```

### 2.3 信息流转规则（铁律）

1. **团队创建**必须由交付总监执行，严禁委派
2. 所有跨成员信息流必须经交付总监中转，成员间不得直连
3. 专业产出必须由对应成员输出后采信，交付总监只做编排与汇编
4. 子任务命名必须使用 Agent ID（如 `software-architect`），不得使用中文名

## 4. 需求优先级框架

### 4.1 P0/P1/P2 三级体系

| 级别 | 定义 | 判定标准 | 例子 |
|------|------|---------|------|
| **P0** | 必须交付，缺一功能不完整 | 核心闭环的必须环节 | 用户注册/登录、对战引擎核心逻辑 |
| **P1** | 应该交付，显著提升体验 | 高频使用的增强功能 | 对战记录、卡组发布 |
| **P2** | 可以交付，锦上添花 | 低频或可选功能 | 闪卡效果、聊天页、社交评论 |

### 4.2 优先级决策矩阵

面对新需求时，按以下维度打分（1-5），总分高者优先：

| 维度 | 权重 | 说明 |
|------|------|------|
| 用户价值 | 40% | 解决了多少用户的什么问题 |
| 技术可行性 | 20% | 现有架构能否支撑，改动范围多大 |
| 业务价值 | 20% | 是否提升留存/活跃/转化 |
| 紧急度 | 20% | 是否有时间敏感性（如新卡包发布配合） |

### 4.3 典型需求优先级判定示例

| 需求 | 用户价值 | 技术可行 | 业务价值 | 紧急度 | 总分 | 级别 |
|------|:--:|:--:|:--:|:--:|:--:|:--:|
| 用户注册登录 | 5 | 4 | 5 | 4 | 4.6 | P0 |
| 赛季排行榜 | 3 | 3 | 3 | 1 | 2.6 | P2 |
| 修复起手抽5张而非6张 | 5 | 5 | 4 | 5 | 4.8 | P0 (Bug) |
| 卡图加载优化 | 2 | 2 | 1 | 1 | 1.6 | P2 |

## 5. 发布管理流程

### 5.1 发布 CheckList

每次发布前必须逐项确认：

#### 代码质量
- [ ] `npm run build` 零错误
- [ ] `npm test` 全部通过（或标注已知遗留问题）
- [ ] 全局一致性审查通过（IS_PASS: YES）
- [ ] TypeScript 严格模式无警告

#### 功能验证
- [ ] 新增功能在真实浏览器中验证通过
- [ ] 原有功能回归无异常
- [ ] 卡牌数据与卡图一致（如有数据变更）
- [ ] 对战引擎完整对局可正常完成

#### 文档同步
- [ ] 变更涉及数据文件时，所有关联文件已同步更新
- [ ] PRD/设计文档已更新（如适用）
- [ ] README/CHANGELOG 已更新

#### 发布操作
- [ ] Git commit 信息清晰描述变更内容
- [ ] 推送到 GitHub main 分支
- [ ] 部署到目标环境（如适用）

### 5.2 版本号规范

```
主版本.次版本.修订号

- 主版本：重大架构变更或不兼容升级
- 次版本：新功能发布（如新页面、用户系统）
- 修订号：Bug 修复、卡效修正、文案调整
```

| 示例 | 含义 |
|------|------|
| `v1.0.0` | 初始发布 |
| `v1.1.0` | 新增用户系统 |
| `v1.1.1` | 修正卡效 Bug |
| `v2.0.0` | 架构重构（如引入后端服务） |

### 5.3 发布频率建议

| 类型 | 频率 | 典型内容 |
|------|------|---------|
| **热修复** | 发现即修，随时发布 | 严重 Bug 修复 |
| **小版本** | 每周 1-2 次 | 卡效修正、UI 微调 |
| **功能版本** | 每月 1 次 | 新页面/新功能模块 |
| **大版本** | 每季度 1 次 | 架构变更/新系统上线 |

## 6. 需求变更与增量开发

### 6.1 变更原则

在已有项目基础上提出变更需求时，遵循最小变更原则：

```
旧 PRD + 新需求 → 增量 PRD（仅描述变更部分）
旧设计 + 增量 PRD → 增量设计 + 增量任务列表
已有代码 + 增量任务 → 修改已有代码 + 新增代码（最小变更）
全量回归测试 + 新功能测试 → QA 验证
```

### 6.2 变更影响评估表

| 变更类型 | 预计影响范围 | 风险评估 | 建议工作流 |
|---------|------------|---------|-----------|
| 新增卡包（无新卡效） | cards.json + 卡图 | 低 | 快速模式 |
| 新增卡包（含新卡效） | cards.json + effects/ + 测试 | 中 | 标准 SOP |
| 修改 UI 布局 | 1-3 个页面组件 | 低 | 快速模式 |
| 修改对战规则 | engine.ts + 测试 | 高 | 标准 SOP |
| 新增页面/功能模块 | 5+ 新文件 + App.tsx | 中-高 | 标准 SOP |
| 修改卡牌数据结构 | 全量类型+数据+UI | 高 | 标准 SOP |

### 6.3 卡牌数据更新提醒

更新卡牌时必须同步以下 **3 个文件**：

1. `public/cards.json` — 项目使用的卡牌数据
2. 外部数据源 `超英击战_卡牌数据.json` — 原始数据源
3. 外部数据源 `超英击战_卡牌数据.xlsx` — 原始数据源

三者必须保持一致。新增卡牌时还需放置卡图到 `public/cards/`。

## 7. 质量标准与验收

### 7.1 代码质量标准

| 维度 | 标准 | 验证方式 |
|------|------|---------|
| **类型安全** | TypeScript strict 模式，零错误 | `npx tsc --noEmit` |
| **构建成功** | Vite build 零错误 | `npm run build` |
| **测试覆盖** | 核心模块有测试，新增卡效至少 1 个测试 | `npm test` |
| **全局一致性** | 所有文件接口一致，无断裂引用 | 工程师全局审查 IS_PASS: YES |
| **代码风格** | 遵循 Google-style，可读可扩展 | Code Review |

### 7.2 功能验收标准

| 功能类型 | 验收标准 |
|---------|---------|
| **新增页面** | 3 种分辨率（1920/1366/375）显示正常；核心交互路径可走通 |
| **新增卡效** | 正常触发 + 边界条件不触发 + 与相关效果交互正确 |
| **用户系统** | 注册→登录→使用→登出全链路；token 过期处理；错误状态 UI |
| **Bug 修复** | 原场景不复现 + 相关场景无回归 |

### 7.3 QA 验收流程

```
1. 工程师交付代码（IS_PASS: YES）
   ↓
2. QA 编写测试用例 + 运行
   ↓
3. 智能路由判定：
   ├── 源码有 Bug → 反馈给工程师修复（最多 2 轮）
   ├── 测试代码有 Bug → QA 自行修复
   └── 全部通过 → 验收通过
```

## 9. 风险管控清单

### 9.1 技术风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|:--:|:--:|------|
| 卡效实现与规则书不一致 | 中 | 高 | 每次新卡包发布后逐卡验证；自动化测试覆盖 |
| 浏览器兼容性问题 | 低 | 中 | 使用标准 Web API；主流浏览器测试 |
| 性能退化（大卡组加载慢） | 低 | 低 | 卡图懒加载；虚拟滚动预留 |
| Supabase 服务中断 | 低 | 中 | 用户系统降级为仅本地模式 |
| TypeScript/依赖版本升级不兼容 | 低 | 中 | 锁版本号；升级前在分支测试 |

### 9.2 产品风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|:--:|:--:|------|
| 新卡包发布延迟 | 中 | 中 | 提前获取卡牌信息，预建框架 |
| 规则书修订导致引擎重构 | 低 | 高 | 引擎架构保持灵活性，参数外置 |
| 竞品出现（其他 Marvel TCG 工具） | 中 | 低 | 差异化：完整的对战引擎 + 用户系统 |
| 版权风险 | 低 | 高 | 标注非官方、MIT 开源、数据由社区维护 |

### 9.3 应急响应

| 问题类型 | 响应时间 | 处理流程 |
|---------|:--:|------|
| 网站无法访问 | 4 小时内 | 检查部署状态 → 回滚 → 修复 → 发布 |
| 严重逻辑 Bug（对战结果错误） | 24 小时内 | BugFix 快捷路径 → 热修复发布 |
| 数据错误（卡牌数据不对） | 48 小时内 | 修正 cards.json → 同步外部数据源 → 发布 |
| 非关键 UI Bug | 下个迭代 | 记录 backlog → 优先级排序 |
